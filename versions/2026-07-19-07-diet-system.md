# Diet system (Food, Meals, Plans, client assignment) + global dialog scroll fix

## Context

The coach CMS has no diet/nutrition management yet — it's the other half of "programming"
alongside Training (which already has muscle groups → categories → exercises → splits, and
split-assignment-to-clients with progress tracking, built in prior rounds). This round adds the
mirror-image system for nutrition: **Food → Meals → Diet Plans**, each level built on the last
exactly like Training's hierarchy, plus assigning a diet plan to a client (mirrors split
assignment, but simpler — see below), plus a macro-recommendation helper used while building a
plan, plus real seed data, plus a global dialog UX fix requested alongside this feature.

Also fixing, app-wide: every dialog currently scrolls its header/footer along with its content
(each form dialog opts into `max-h-[90vh] overflow-y-auto` individually). The user wants the
header and footer fixed and only the middle content scrollable, in every dialog — old and new.

## Global dialog fix (do this first — new diet dialogs should be built against the fixed version)

`src/components/ui/dialog.tsx`: make this the default behavior of the shared primitives instead of
a per-dialog opt-in.
- `DialogContent`: add `max-h-[85vh] overflow-y-auto` to its base className (already the exact
  pattern 6 dialogs currently bolt on individually — `ProductFormDialog`, `ExerciseFormDialog`,
  `CollectionFormDialog`, `MuscleCategoryFormDialog`, `SplitFormDialog`,
  `ClientProfileFormDialog`). `cn()` uses `tailwind-merge` (confirmed in `src/lib/utils.ts`), so
  this is safe to add to the base without fighting per-instance classes.
- `DialogHeader`: add `sticky top-0 z-10 bg-background pb-4 border-b border-border` — `position:
  sticky` inside the now-scrolling `DialogContent` keeps it pinned at the top; the solid
  background masks content scrolling underneath.
- `DialogFooter`: add `sticky bottom-0 z-10 bg-background pt-4 border-t border-border` — same idea,
  pinned to the bottom. Works regardless of `DialogFooter` being nested inside each dialog's
  `<form>` (sticky only needs the scrolling ancestor in its containment chain, not to be a direct
  child of it).
- `DialogPrimitive.Close` (the "X" button): bump to `z-20` so it stays above the new sticky header
  bar's `z-10` stacking context.
- Clean up the 6 existing dialogs: strip their now-redundant `max-h-[90vh] overflow-y-auto`
  className fragments (keep whatever `max-w-*` they each set — that's a legitimate per-dialog
  size, not part of this fix).

No dialog's internal JSX structure needs to change (Header/form/Footer nesting stays exactly as
it is) — this is a two-file-pattern change (1 primitive + 6 call-site cleanups), not a per-dialog
rewrite.

## Data model (four new MongoDB collections)

```ts
// foods — nutrition facts are always "per 100g", same convention as a standard nutrition label
{
  _id: ObjectId;
  name: string;
  image: string | null;
  caloriesPer100g: number;
  carbsPer100g: number;
  proteinPer100g: number;
  fatPer100g: number;
  createdAt: Date;
  updatedAt: Date;
}

// meals
{
  _id: ObjectId;
  name: string;
  foods: Array<{ foodId: ObjectId; quantityGrams: number }>;
  createdAt: Date;
  updatedAt: Date;
}

// dietPlans — always exactly 7 day entries, same invariant as splits
{
  _id: ObjectId;
  name: string;
  description: string;
  days: Array<{
    day: "monday" | ... | "sunday";
    label: string;              // optional, e.g. "High Carb Day"
    mealIds: ObjectId[];        // meals assigned that day, duplicates allowed (e.g. same snack twice)
  }>;
  createdAt: Date;
  updatedAt: Date;
}

// dietPlanAssignments — append-only history, mirrors splitAssignments
{
  _id: ObjectId;
  clientId: ObjectId;
  dietPlanId: ObjectId;
  startDate: Date;
  createdAt: Date;
}
```

**No duration/progress-bar for diet plan assignment**, unlike splits — a diet plan isn't a
time-boxed program the way a split is (splits explicitly have `durationWeeks`; the user didn't
ask for one on diet plans). "Current vs. history" stays the same derived pattern as splits (latest
`startDate` wins), but the client detail page shows "Assigned since {date}" instead of a progress
bar, and `AssignDietPlanDialog` is the *simple* version of `AssignSplitDialog` — split/date picker,
no in-progress-replacement prompt, because there's no progress to discard or keep.

**Referential integrity** (same pattern as `categories.server.ts`/`splits.server.ts`): deleting a
food is blocked if any meal references it; deleting a meal is blocked if any diet plan references
it; deleting a diet plan is blocked if any client still has it assigned.

## Nutrition math (`src/diet/nutrition.ts` — pure, no I/O, mirrors `src/clients/progress.ts`)

- `foodNutritionForQuantity(food, quantityGrams)` and summing helpers for a meal's/day's totals —
  meals/diet-plan-days are resolved server-side with denormalized names (same pattern as
  `SplitExercise.exerciseName`), and the coach UI already has the full `Meal[]` list in memory
  (fetched for the meal picker), so day/plan totals are computed **entirely client-side** by
  looking up each assigned meal locally — no extra RPC needed.
- `recommendMacros(profile: {weight, height, dob}, goal: "bulk" | "cut"): MacroTarget | null` — the
  macro helper tool's core. Returns `null` if the client has no weight/height set yet (nothing to
  compute from). This app's `clientProfiles` doesn't collect sex, so this uses a **sex-neutral
  approximation**, clearly commented as such:
  - BMR ≈ `10×weight(kg) + 6.25×height(cm) − 5×age − 78` (Mifflin-St Jeor, averaging the male
    `+5` and female `−161` constants into `−78`; age from `dob`, defaulting to 30 if unset)
  - Maintenance = BMR × 1.55 (a "moderately active" multiplier — a reasonable default for a
    coached client; no activity-level field exists to ask instead)
  - Target calories = maintenance × 0.8 (cut, 20% deficit) or × 1.15 (bulk, 15% surplus)
  - Protein = 2.2g/kg bodyweight (both goals — preserves/builds muscle either way); Fat = 25% of
    target calories; Carbs = whatever's left
  
  This is a reasonable, standard-formula estimate, not a medical/dietitian-grade calculation — the
  UI will label it as such. Flagging this now since it's the one part of this plan that's an
  assumption rather than a direct mirror of an existing pattern; happy to adjust the constants if
  you'd rather.

## Server layer (`src/diet/`, mirrors `src/training/`)

`types.ts`, `nutrition.ts` (above), `foods.server.ts`, `meals.server.ts` (resolves each food's
name + per-100g macros into `Meal.foods` — same denormalization `exercises.server.ts` does for
`muscleGroupName`), `diet-plans.server.ts` (resolves `mealId → mealName`, same as
`splits.server.ts`), `diet-assignments.server.ts` (`listDietPlanAssignmentsForClient`,
`assignDietPlan` — validates target user is `role: "client"` and the plan exists, same guard as
`assignments.server.ts`'s `assignSplit`), `upload.server.ts` (`saveFoodImage`, thin wrapper over
the existing generic `saveAsset(file, "foods", IMAGE_MIMES)`), `functions.ts` (RPCs, all
`coachMiddleware`).

**Wiring into Clients**: `src/clients/types.ts` gains `currentDietPlan: DietPlanAssignment | null`
on `ClientListItem` and `dietPlanHistory: DietPlanAssignment[]` on `ClientDetail`.
`src/clients/clients.server.ts`'s `listClients`/`getClientDetail` call the new
`listDietPlanAssignmentsForClient` (imported from `@/diet/diet-assignments.server`) exactly the
way they already call `listAssignmentsForClient` for splits — same shape, one more field.

## Coach CMS UI

- **`/coach/diet`** (new route, `src/routes/coach/diet.tsx`): three tabs — Food, Meals, Plans —
  same `Tabs` + per-tab `useQuery`/`useMutation` pattern as `/coach/training`. Nav item added to
  `coach.tsx`'s `NAV_ITEMS` (real, not "Uncompleted"), `Utensils` icon, placed after Training.
- **Food tab**: table (image thumbnail, name, calories, actions) + `FoodFormDialog` (name, single
  image upload — same blob-preview pattern as `MuscleGroupFormDialog.tsx` — and four number
  inputs for calories/carbs/protein/fat per 100g).
- **Meals tab**: table (name, food count, total calories, actions) + `MealFormDialog` (name, then
  repeatable food rows — `Select` a food + grams quantity + remove button + "Add Food", same
  repeatable-row shape as `SplitFormDialog`'s exercise rows) with a **live nutrition total**
  readout at the bottom (calories/carbs/protein/fat), computed via `nutrition.ts` as rows change —
  same live-readout idea as `ProductVariantFields.tsx`'s "price after discount".
- **Plans tab**: table (name, day count with meals, actions) + `DietPlanFormDialog` (name,
  description, then a 7-day `Accordion` — identical structure to `SplitFormDialog`: each day has a
  label input + repeatable meal-picker rows). Each day shows its own **live nutrition total**
  (same computation, summed across that day's meals). Above the accordion, the **macro helper
  tool**: a `Select` over `getCoachClientsFn()` results + a Bulk/Cut `Select`; once both are
  picked, `recommendMacros()` runs against that client's profile and each day's live total gets an
  inline delta ("+310 kcal over target" / "−18g protein under target") so the coach can adjust
  that day's meals before saving — directly satisfies "before assigning it to the day ... before
  assigning it to the client." This is display-only (no server round-trip, no persisted
  "recommended for" field on the plan) — purely a build-time aid.
- **Client detail page** (`src/routes/coach/clients_.$clientId.tsx`): a third card, "Current Diet
  Plan" (plan name, "Assigned since {date}", **Assign Diet Plan** button →
  `AssignDietPlanDialog.tsx` — plan `Select` + start-date picker, mirrors the original simple
  `AssignSplitDialog` shape before the in-progress/replace feature existed) plus a "Diet Plan
  History" table below the existing Split History table, same row shape minus the status badge
  (no progress concept to show).

## Seed data

- **`scripts/seed-diet.ts`** (`npm run db:seed:diet`, run after `db:seed:clients`): ~20 foods
  covering the requested staples (rice, chicken breast, peanut butter, eggs, tuna, honey, banana,
  oats, salmon, broccoli, sweet potato, almonds, greek yogurt, olive oil, whole wheat bread, milk,
  lean beef, avocado, spinach, apple) with real approximate per-100g macros (standard USDA-ish
  values — noted in the script as approximate, not medical-grade). ~8 meals built from them
  (e.g. "Chicken & Rice Bowl", "Protein Oats", "PB Banana Toast", "Tuna Salad", "Egg Breakfast",
  "Salmon & Sweet Potato", "Greek Yogurt Snack", "Beef & Rice"). 3 diet plans, each assigning a
  different subset of those meals to all 7 days ("Lean Bulk Plan" ~4 meals/day, "Cutting Plan" ~3
  leaner meals/day, "Maintenance Plan" ~3 moderate meals/day) — foods/meals/plans idempotent-vs-
  full-replace following the exact same convention as `seed-training.ts` (muscle-group-style name
  matching isn't needed here since there's no cross-script dependency on foods by name from
  elsewhere; plans and meals are demo data, cleared and reinserted fresh each run, same reasoning
  as `seed-training.ts`'s exercises/splits).
- **Assignment "based on height and weight"**: the script duplicates the small `recommendMacros`-
  style maintenance-calorie formula (standalone, same reasoning `seed-clients.ts` already
  duplicates bcrypt/upsert logic rather than importing `src/`) to compute each seeded client's
  maintenance calories from their real seeded weight/height, then assigns whichever of the 3 plans
  has the closest actual average daily calories (computed from the real seeded food data, not
  hardcoded) — genuinely data-driven, not a hardcoded lookup table. Jordan Ellis additionally gets
  an earlier diet-plan history entry (a different plan) so the Diet Plan History table has
  something to show, mirroring Jordan's split history. Morgan Blake (no profile) gets no diet
  assignment, consistent with their already-established fully-empty state.
